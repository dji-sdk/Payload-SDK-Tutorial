## Introduction
This article provides a quick-start demo based on the **Raspberry Pi 4B development board**, guiding developers through the complete process—from hardware connection and PSDK environment setup to sample program configuration and demo validation—in the shortest possible time.

## What You Will Achieve with This Demo
After completing this quick-start demo, you will be able to:
- Use a **Raspberry Pi 4B** as an external Linux payload device.  
- Connect the Raspberry Pi development board to the flight platform via the **E‑Port V2 / E‑Port Development Board**.  
- Compile and run PSDK sample programs on the Raspberry Pi.  
- Verify payload recognition and basic interaction functions in **DJI Pilot**.

## Basic Knowledge Overview
The following is an introduction to the hardware devices required for this tutorial. It is recommended to review the related materials before proceeding with the subsequent sections.
* [Raspberry Pi 4B Developer Kit](https://www.raspberrypi.com/products/raspberry-pi-4-model-b/)

## Hardware Environment Setup
### Device Connection
#### Matrice 4D / 4TD Device Connection
##### Device Connection via E-Port

As depicted in the *Figure. Raspberry Pi 4B* and *Figure. Raspberry Pi 4B Developer Board Connection*, connect the E-Port Developer Kit to the Raspberry Pi 4B Developer Board. For details about the 40-pin GPIO on the developer board, see the [Pin Introduction](https://www.raspberrypi.com/documentation/computers/raspberry-pi.html).

1. Use dupont lines to connect the UART (Label 3) on the E-Port Developer Kit to the USB to TTL Serial Module (TX, RX, and GND of the Developer Kit to RX, TX, and GND of the serial module respectively).
2. Connect the USB to TTL Serial Module to the USB 3.0 port (Label f) of the Raspberry Pi 4B Developer Board.
3. Set the USB host/device toggle switch (Label 4) on the E-Port Developer Kit to the **Host** position.
4. Using the USB-C OTG adapter and USB-C cable, connect the USB (Label 5) on the E-Port Developer Kit to the USB-C (Label a) on the developer board.
5. Check and confirm that the power DIP switch (Label 1) on the E-Port Development Kit is set to the **OFF** position.
6. Connect the USB port (Label 2) on the E-Port Development Kit to the aircraft’s E-Port port using the coaxial cable. Make sure to note the correct **A/B side orientation**.
7. Power on the aircraft. Then switch on the power DIP switch on the E-Port development board, check whether the green indicator LED is illuminated, and verify that the development board outputs video correctly on the monitor to confirm that it has entered the boot page.

#### Matrice 400 Device Connection
1. Set the **ENABLE** switch on the E‑Port V2 development board to **ON**, and set the **SWITCH** to the **EXPAND** position. Use the E‑Port V2 coaxial cable to connect the board to the aircraft’s E‑Port V2 port. It is currently recommended to use **Port 1**.
2. Flash the Raspberry Pi 4B development board with the official operating system image (x64 version), then connect the Raspberry Pi 4B to its power supply port using a USB cable (which also provides USB 2.0 communication capability).
3. Power on the aircraft and check whether the green power indicator LED on the E‑Port V2 development board is illuminated, and verify that the Raspberry Pi has booted properly.

:::tip Note:
When developing PSDK using E‑Port V2 and configuring the PSDK link as `DJI_USE_ONLY_UART`, the available VCOM or USB serial port combinations must be selected from the following options:
* USB serial adapter with **CP2101** as the main chip, compliant with USB 2.0, VID:PID **10C4:EA60**
* USB serial adapter with **FT232T** as the main chip, compliant with USB 2.0, VID:PID **0403:6001**
* Implement USB VCOM on an external device with VID:PID set to **2CA3:F002**
:::

### Authentication Chip Integration

The DJI Authentication Chip is used to authenticate third-party payload devices developed based on the DJI E‑Port / E‑Port Lite port. It ensures that the payload device can operate properly and communicate effectively with the aircraft, providing payload security authentication and function authorization.  
For detailed specifications of the Authentication Chip, refer to the <a href="../configure-the-hardware-environment/sdk-cc.html" target="_blank">Authentication Chip Integration</a> documentation.

::: warning Note
- **Matrice 4D / 4TD** aircraft **must integrate the Authentication Chip**; otherwise, **PSDK functions will not be available**.
- When **Matrice 400** is used with the **E‑Port V2 Development Kit** (E‑Port V2 coaxial cable / SkyPort V3 adapter ring), an encryption/authentication chip is already integrated inside the kit. **Developers do not need to handle I²C read/write operations** for the chip and only need to complete the **binding and unbinding** procedures as specified.
:::

The DJI Authentication Chip is connected via an adapter board. Power is supplied to the chip through the expansion I/O pins of the Raspberry Pi 4B, and the I²C communication lines and the RST reset pin are connected accordingly.

#### Pin Connection Description
The sample code uses the I²C device `/dev/i2c-1`. The pin connections are as follows:

| Authentication Chip Pin | Raspberry Pi 4B GPIO |
|:-----------------------:|:--------------------:|
| SCL                     | GPIO3                |
| SDA                     | GPIO2                |
| RST                     | GPIO4                |

#### Recommended Integration Method for the Debug Phase
During the debugging and validation phase, it is recommended to integrate the Authentication Chip using the following method:
- Place the Authentication Chip in a DFN8 (2×3) programming adapter socket.
- Connect the adapter socket to the Raspberry Pi GPIO pins using Dupont wires.
- Example adapter socket model: **05D4 DFN8 2×3** (can be purchased separately).

::: warning Note
You can use the `i2cdetect` tool to scan connected I²C devices and quickly verify whether the Authentication Chip is operating correctly. If the device is not detected, try reducing the I²C clock frequency and scan again.
:::

1. The Raspberry Pi 4B Developer Kit does not include an SD card and Raspberry Pi OS 64 bit system image by default. Follow the [Raspberry Pi Imager](https://www.raspberrypi.com/documentation/computers/getting-started.html#using-raspberry-pi-imager) for image programming and account setup.

3. On the desktop, verify the Raspberry Pi 4B's network connectivity using any browser webpage.
4. Before running PSDK, install the following dependencies using the shell commands provided. Ensure successful installation.
   - FFmpeg
   - ffplay
   - OpenCV
   - libaio
   - CMake
   - libusb

   /* Update software repository */

### 2. Enable Raspberry Pi 4B's USB device function

**Scenario 1: Developing SDK Device Using EPort port**

You can refer to the [Raspberry Pi bulk endpoint configuration guide](https://sdk-forum.dji.net/hc/zh-cn/articles/10232604141465-M30%E5%BC%80%E5%8F%91-%E6%A0%91%E8%8E%93%E6%B4%BE4B%E9%85%8D%E7%BD%AEUSB-device-RNDIS-%E5%92%8C-BULK) to enable the USB device functionality on the Raspberry Pi 4B.

**Scenario 2: Developing SDK Device Using E-Port V2**
> This setting only needs to be configured on the Matrice 400 model.

You can refer to the [Raspberry Pi bulk endpoint configuration guide](https://sdk-forum.dji.net/hc/zh-cn/articles/10232604141465-M30%E5%BC%80%E5%8F%91-%E6%A0%91%E8%8E%93%E6%B4%BE4B%E9%85%8D%E7%BD%AEUSB-device-RNDIS-%E5%92%8C-BULK) to enable the USB device functionality on the Raspberry Pi 4B.

Please note that you need to replace `raspi-usb-device-start.sh` with the following content.
When developing SDK devices using E-Port V2, the Raspberry Pi can only function as either a VCOM, RNDIS, or USBBULK device at the same time.

> When using BULK functionality, set VID:PID to 2CA3:F001.
> When using VCOM functionality, set VID:PID to 2CA3:F002.
> When using RNDIS functionality, set VID:PID to 2CA3:F003.

```shell
#!/bin/bash

# The IP address shared by all USB network interfaces created by this script.
net_ip=192.168.42.120
# The associated netmask.
net_mask=255.255.0.0

enable_vcom=0
enable_rndis=0
enable_bulk=1

startup_bulk_dir=$(dirname "$0")
startup_bulk_exe=${startup_bulk_dir}/build/startup_bulk

if [ ! -d /sys/kernel/config/usb_gadget ]; then
    if ! $(grep -q dtoverlay=dwc2 /boot/config.txt) ; then
        echo -e "\n\n\n## customized\n# for op-sdk enable USB device RNDIS\n[all]\ndtoverlay=dwc2" >> /boot/config.txt
        init 6
    fi

    if ! $(grep -q modules-load=dwc2 /boot/cmdline.txt) ; then
        sed -i 's/rootwait /&modules-load=dwc2 /' /boot/cmdline.txt
        init 6
    fi

    if ! $(grep -q libcomposite /etc/modules) ; then
        echo -e "\n\n\n## customized\n# for op-sdk enable USB device RNDIS\nlibcomposite" >> /etc/modules
        init 6
    fi

    exit 1
fi

modprobe g_ether
rmmod g_ether
sleep 1

if [ ! -d /sys/kernel/config/usb_gadget/pi4 ]; then
    mkdir -p /sys/kernel/config/usb_gadget/pi4
    cd /sys/kernel/config/usb_gadget/pi4
        echo 0x2ca3 > idVendor # Linux Foundation
        echo 0xF001 > idProduct # Multifunction Composite Gadget

        if [ ${enable_vcom} -eq 1 ]; then
            echo 0xF002 > idProduct # Multifunction Composite Gadget
        fi

        if [ ${enable_rndis} -eq 1 ]; then
            echo 0xF003 > idProduct # Multifunction Composite Gadget
        fi

        if [ ${enable_bulk} -eq 1 ]; then
            echo 0xF001 > idProduct # Multifunction Composite Gadget
        fi

        echo 0x0001 > bcdDevice # v1.0.0
        echo 0x0200 > bcdUSB # USB2

        echo 0xEF > bDeviceClass
        echo 0x02 > bDeviceSubClass
        echo 0x01 > bDeviceProtocol

        mkdir -p strings/0x409
            echo "psdk-rpi4b" > strings/0x409/serialnumber
            echo "raspberry" > strings/0x409/manufacturer
            echo "PI4" > strings/0x409/product

        cfg=configs/c.1

        mkdir -p "${cfg}"
            echo 0x80 > ${cfg}/bmAttributes
            echo 250 > ${cfg}/MaxPower

        cfg_str=""

        if [ ${enable_rndis} -eq 1 ]; then
            cfg_str="${cfg_str}+RNDIS"
            func=functions/rndis.usb0
            mkdir -p "${func}"
                ln -sf "${func}" "${cfg}"

            # Informs Windows that this device is compatible with the built-in RNDIS
            # driver. This allows automatic driver installation without any need for
            # a .inf file or manual driver selection.
            echo 1 > os_desc/use
            echo 0xcd > os_desc/b_vendor_code
            echo MSFT100 > os_desc/qw_sign
                echo RNDIS > "${func}/os_desc/interface.rndis/compatible_id"
                echo 5162001 > "${func}/os_desc/interface.rndis/sub_compatible_id"

            ln -sf "${cfg}" os_desc
        fi

        if [ ${enable_vcom} -eq 1 ]; then
            cfg_str="${cfg_str}+ACM0"
            func=functions/acm.gs0
            mkdir -p "${func}"
            ln -sf "${func}" "${cfg}"
        fi

        if [ ${enable_bulk} -eq 1 ]; then
            mkdir -p /dev/usb-ffs

            cfg_str="${cfg_str}+BULK1"
            mkdir -p /dev/usb-ffs/bulk1
            func=functions/ffs.bulk1
            mkdir -p "${func}"
            ln -sf "${func}" "${cfg}"
            mount -o mode=0777,uid=2000,gid=2000 -t functionfs bulk1 /dev/usb-ffs/bulk1

            cfg_str="${cfg_str}+BULK2"
            mkdir -p /dev/usb-ffs/bulk2
            func=functions/ffs.bulk2
            mkdir -p "${func}"
            ln -sf "${func}" "${cfg}"
            mount -o mode=0777,uid=2000,gid=2000 -t functionfs bulk2 /dev/usb-ffs/bulk2

            cfg_str="${cfg_str}+BULK3"
            mkdir -p /dev/usb-ffs/bulk3
            func=functions/ffs.bulk3
            mkdir -p "${func}"
            ln -sf "${func}" "${cfg}"
            mount -o mode=0777,uid=2000,gid=2000 -t functionfs bulk3 /dev/usb-ffs/bulk3
        fi

        mkdir -p "${cfg}/strings/0x409"
        echo "${cfg_str:1}" > "${cfg}/strings/0x409/configuration"
    # cd -

fi

if [ ${enable_bulk} -eq 1 ]; then
    if [ ! -f ${startup_bulk_exe} ]; then
        cd ${startup_bulk_dir}
            make && ( $? != 0 ) && ( exit 1 )
            cp startup_bulk ${startup_bulk_exe}
            make clean
        cd -
    fi

    ${startup_bulk_exe} /dev/usb-ffs/bulk1 &
    sleep 1
    ${startup_bulk_exe} /dev/usb-ffs/bulk2 &
    sleep 1
    ${startup_bulk_exe} /dev/usb-ffs/bulk3 &
    sleep 1
fi

    udevadm settle -t 5 || :
    ls /sys/class/udc > UDC

if [ ${enable_rndis} -eq 1 ]; then
    /sbin/brctl addbr pi4br0
    /sbin/ifconfig pi4br0 ${net_ip} netmask ${net_mask} up

    /sbin/brctl addif pi4br0 usb0
    /sbin/ifconfig usb0 down
    /sbin/ifconfig usb0 up
fi
exit 0

```

### SDK Authentication Chip Software Initialization

Developers must call the `DjiPlatform_RegHalI2cHandler` API to register the implementation of a standard I2C device driver. During the PSDK initialization process, the SDK automatically completes the authentication procedure of the SDK Authentication Chip through this callback interface. For a reference implementation, see `samples/sample_c/platform/linux/raspberry_pi/hal/hal_i2c.c`.

```
    returnCode = DjiPlatform_RegHalI2cHandler(&i2CHandler);
    if (returnCode != DJI_ERROR_SYSTEM_MODULE_CODE_SUCCESS) {
        printf("register hal i2c handler error");
        return DJI_ERROR_SYSTEM_MODULE_CODE_SYSTEM_ERROR;
    }
```

### Compilation and Execution

The PSDK software package supports CMake compilation. Use the **cmake** command to compile and generate executable files. Executing the example code below creates executable files `dji_sdk_demo_on_jetson` and `dji_sdk_demo_on_jetson_cxx` in the `build/` directory. Raspberry Pi 4B Developer Kit can use the executable files for the Jetson Nano Developer Kit.

```c
cd Payload-SDK/
mkdir build && cd build
cmake ../  && make -j4
```
### SDK Authentication Chip Binding

1. After successful compilation, run the PSDK program. If the Authentication Chip hardware is connected correctly, the serial number (SN) of the Authentication Chip can be obtained successfully. At this point, the log will display the message **"SDK accessory not bound"**.
2. Log in to your developer account using DA2. In DA2, go to the **Payload SDK** tab and check whether the PSDK device can be correctly recognized. Once the device is recognized, click the **Bind** button to perform the binding operation.
3. After the binding is completed successfully, the PSDK device can operate normally.

## FAQ

### Frequently Asked Questions

1. Error when running PSDK:
```c
Can't operation the device. Probably the device has not operation permission. Please execute command 'sudo chmod 777 /dev/ttyUSB0' to add permission.
```
Solution:
Run **sudo chmod 777 /dev/ttyUSB0**

2. Error when running PSDK:
```c
Please fill in correct user information to 'samples/sample_c/platform/linux/nvidia_jetson/application/dji_sdk_app_info.h' file.
```
Solution: `dji_sdk_app_info.h`Please create an account on developer website, and fill in the relevant information to file `dji_sdk_app_info.h`.

3. Error when running PSDK:：
```c
ls: cannot access '/dev/ttyUSB0': No such file or directory
```
Solution: Check if the serial port is connected correctly.
