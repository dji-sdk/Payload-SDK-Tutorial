## DPK Application Management Tool

The `dji_app_ctl` tool can be used in the CLI environment to install, update, uninstall, stop, and start applications. It can also be used to view application information and status.

:::warning Note:
- To install a DPK package using this tool, make sure that Manifold 3 is properly connected to the aircraft and that the aircraft is powered on.
- This tool is currently intended only for development and debugging. Its startup and runtime mechanism differs from the application management mechanism in DJI Pilot / FlightHub 2. Before releasing your application to a production environment, be sure to test application installation and startup using DJI Pilot / FlightHub 2, especially if your application requires high-performance mode (all 8 CPU cores enabled).
:::

- **Install / update an application**

```bash
dji_app_ctl install -i xxx.dpk
```

- **Uninstall an application**

```bash
dji_app_ctl uninstall xxx(app_name)
```

- **Start an application**

```bash
dji_app_ctl start xxx(app_name)
```

- **Stop an application**

```bash
dji_app_ctl stop xxx(app_name)
```

- **Restart an application**

```bash
dji_app_ctl restart xxx(app_name)
```

- **Enable application auto-start at boot**

```bash
dji_app_ctl enable xxx(app_name)
```

- **Disable application auto-start at boot**

```bash
dji_app_ctl disable xxx(app_name)
```

- **View information for all applications**

```bash
dji_app_ctl list
```

- **View runtime status for all applications**

```bash
dji_app_ctl status
```

- **View information for all applications with the specified application name**

```bash
dji_app_ctl list xxx(app_name)
```

- **View runtime status for all applications with the specified application name**

```bash
dji_app_ctl status xxx(app_name)
```

## Network Configuration Tool

The expansion port on Manifold 3 supports external USB Ethernet adapters, USB Wi‑Fi adapters, and 4G/5G CPE devices to meet diverse network access requirements. Through the `netctl.sh` script tool, network interfaces can be managed in a unified manner, including static IP configuration, Wi‑Fi connection, and routing priority settings for multiple network adapters.

### Basic Description

`netctl.sh` is a network configuration script tool based on NetworkManager (`nmcli`) for device-level network configuration on Manifold 3. It mainly supports the following:

- Wi‑Fi connection and disconnection (Open / WPA2 / WPA3-PSK)
- Static IP and DHCP configuration for Wi‑Fi / wired networks
- Declarative routing and multi-uplink priority configuration based on NetworkManager (supports automatic failover when a link becomes unavailable)

:::warning Note:
This tool does not handle any authentication process that requires human interaction. It is only responsible for network device-level configuration.
:::
#### Basic Usage

```bash
netctl.sh <command> [options]
```

#### Command Overview

- **Scan Wi‑Fi**

```bash
netctl.sh scan_wifi [--iface IFACE]
```

- **Connect to Wi‑Fi**

```bash
netctl.sh connect_wifi --ssid <SSID> [--password <PSK>] [--iface IFACE]
```

- **Disconnect Wi‑Fi**

```bash
netctl.sh disconnect_wifi --ssid <SSID> [--iface IFACE]
```

- **Configure a static IP**

```bash
netctl.sh setup_static_ip --type <TYPE> [--ssid <SSID>] [--iface IFACE] --ip <IP> --gw <Gateway> [--mask <CIDR>] [--dns DNS1,DNS2]
```

- **Restore DHCP mode**

```bash
netctl.sh setup_dhcp_ip --type <TYPE> [--ssid <SSID>] [--iface IFACE]
```

- **Configure routes**

```bash
netctl.sh setup_route --route "NET>TYPE" [--route "..."] [--iface TYPE:IFACE]
```

- **Reset route configuration**

```bash
netctl.sh setup_route --reset
```

- **View brief help**

```bash
netctl.sh -h
netctl.sh --help
```

- **View full help**

```bash
netctl.sh --help-full
```

#### Recommended Network Adapters

For USB Wi‑Fi adapters, **RTL8852BU** and **RTL88X2BU** are recommended. Drivers for these two models are preinstalled on Manifold 3.   
If you use other models, make sure that the selected adapter has a supported driver available.

#### `--iface` Description (Multiple Network Adapters and Interface Selection Strategy)
:::tip
It is recommended to use the `ifconfig` command to confirm the actual names of network interfaces before specifying them with this parameter, in order to avoid execution failure of `netctl.sh` caused by an incorrect interface name.
:::

- If the specified `TYPE` corresponds to only one available interface, the system will automatically select that interface and `--iface` does not need to be specified.
- If the specified `TYPE` corresponds to multiple available interfaces, the interface must be explicitly specified using `--iface`. If `--iface` is not specified when multiple interfaces exist, the command will report an error and exit directly.

### Authentication and Security Boundary Description

This tool is a device-level network configuration tool, not a user session management tool. When connecting to Wi‑Fi:

✅ **Supported:**
- Open Wi‑Fi
- WPA2 / WPA3-PSK
- Direct Ethernet connection

❌ **Not supported:**
- Captive Portal / Web login
- SMS / one-time password
- 802.1X (EAP / PEAP / TLS / TTLS)

Whether re-authentication is required after disconnection or reboot depends entirely on the network's own policy. `netctl.sh` does not control this behavior.

  - If an open Wi‑Fi network is detected, it will be connected directly.
  - If the Wi‑Fi network is encrypted, the system will securely prompt for a password.

- After a successful connection:
  - A NetworkManager connection profile will be created or reused.
  - Auto-reconnect will be enabled, including after reboot.

##### Internet Access Priority Description (Important)

The `connect_wifi` command is only responsible for establishing the Wi‑Fi connection. It does not automatically modify the system route or internet access priority.

If the system is simultaneously connected to Onboard (aircraft network) + Ethernet (wired network) + Wi‑Fi (wireless network), Ethernet and onboard routes have higher priority than Wi‑Fi by default.

Therefore, even if Wi‑Fi is connected successfully, internet traffic may still go through the onboard network or Ethernet. If you want internet access to go through Wi‑Fi, run the `netctl.sh setup_route` subcommand after the Wi‑Fi connection is established to explicitly configure route priority.

- Whether the Wi‑Fi interface is in the `disconnected` state
- Whether there is still a valid IP address or default route

### IP Configuration

#### Configure a Static IP

```bash
netctl.sh setup_static_ip --type <TYPE> [parameters...]
```

- **Wi‑Fi**

```bash
netctl.sh setup_static_ip --type wifi [--iface IFACE] --ssid <SSID> --ip <IP> --gw <Gateway> [--mask <CIDR>] [--dns DNS1,DNS2]
```

- **Ethernet / Cellular**

```bash

```

**Parameter Description**

| Parameter | Required | Description | Example |
| :-- | :-- | :-- | :-- |
| `--type` | Yes | Interface type: `wifi` / `eth` / `cellular` | `--type wifi` |
| `--ssid` | Required for Wi‑Fi scenarios (`--type wifi`) | Wi‑Fi SSID name, used to configure the IP for the connection corresponding to that SSID. Required only for Wi‑Fi scenarios, not for wired scenarios. | `--ssid Airport_WiFi` |
| `--iface` | No | Specifies the interface.<ul><li>If `TYPE` has only one interface, it will be used automatically.</li><li>If multiple interfaces exist, `--iface` must be used to specify one.</li></ul>**Note:** It is recommended to use the `ifconfig` command to confirm the actual names of the network interfaces before specifying one with this parameter, so as to avoid `netctl.sh` execution failure caused by an incorrect interface name.  | `--iface wlan0`<br>`--iface eth0` |
| `--ip` | Required for static IP scenarios | Static IP address. If omitted, the IP will not be changed (the current DHCP or static IP will continue to be used). | `--ip 192.168.1.100` |
| `--mask` | Recommended for static IP scenarios | Prefix length of the subnet mask | `--mask 24` |
| `--gw` | Required for static IP scenarios | Default gateway (route IP). Must be used together with `--ip`. | `--gw 192.168.1.1` |
| `--dns` | No | Comma-separated list of DNS servers. If not specified, the existing DNS settings will be retained; if specified, they will be overwritten. | `--dns 4.4.4.4,8.8.8.8` |

:::warning 

The script does not automatically validate or correct gateway validity when configuring routes or network access. Users must ensure that the configuration is correct. When configuring a static IP, the following rules must be satisfied; otherwise, the configured IP may result in loss of network connectivity.
:::

- **IP address (`--ip`)**
  - Must be within the current LAN subnet.
  - Must not conflict with the IP of any other device on the network.
  - Must not use a network address or broadcast address.

- **Subnet mask (`--mask`)**
  - Use CIDR notation.
  - Common values:
    - `24` → `255.255.255.0` (most common)
    - `16` → `255.255.0.0`
    - `8` → `255.0.0.0`

- **Default gateway (`--gw`, very important)**
  - The gateway must:
    - Be in the same subnet as `--ip`
    - Actually exist and be reachable
  - Typical examples:
    - Home / office network: router IP (e.g. `192.168.1.1`)
    - Industrial network: upstream switch or core router IP

**Behavior Description**

- Only the `ipv4` configuration in NetworkManager is modified.
- If the current connection is in the `active` state, the script will disconnect and then reconnect the network so that the configuration takes effect immediately.
- If the connection is not active, the configuration will take effect the next time the connection is activated.

### Important Limitations of `onboard` Routing

In the Manifold 3 system:

- `onboard` is a black-box egress
- NAT / forwarding is performed by the aircraft
- There is no visible Layer-3 gateway on the device side

Therefore, the following mandatory rules apply:

✅ **Supported:**

```bash
netctl.sh setup_route --route "default>onboard"
```

❌ **Not supported (will be ignored):**

```bash
netctl.sh setup_route --route "1.1.1.1>onboard"
netctl.sh setup_route --route "192.168.140.0/24>onboard"
```

This is a limitation of Linux routing semantics and is considered **intended behavior rather than a defect**.

**Examples**

- All traffic goes through onboard

```bash
netctl.sh setup_route --route "default>onboard"
```

- The target network `10.0.0.0/8` goes through Ethernet, `4.4.4.4` goes through cellular, `192.168.42.0/24` goes through Wi‑Fi, and all remaining target networks go through the onboard aircraft network

```bash
netctl.sh setup_route --route "10.0.0.0/8>eth" --route "4.4.4.4>cellular" --route "192.168.42.0/24>wifi" --route "default>onboard"
```

- For multiple Wi‑Fi adapters, specify `wlan1` as the egress

```bash
netctl.sh setup_route --iface wifi:wlan1 --route "default>wifi"
```

- For a multi-uplink system, bind interfaces separately

```bash
netctl.sh setup_route --iface eth:eth1 --iface wifi:wlan0 --route "default>eth" --route "default>wifi"
```

**Behavior Description**

- Based on declarative routing in NetworkManager.
- Interfaces of the same type use the same `metric`.
- When an egress interface is disconnected, its route becomes invalid automatically, and other egress interfaces take over automatically.

#### Reset Routes

```bash
netctl.sh setup_route --reset
```

**Behavior Description**

- Removes only the routes added by `netctl.sh`.
- Restores the default `metric` configuration of NetworkManager.
- Does not affect system routes not managed by `netctl`.

### Internet Access Priority and Egress Selection

When multiple `TYPE` route egresses exist for the same destination network, the system automatically determines the internet access egress based on the route `metric`:

- The smaller the `metric`, the higher the priority.
- When a higher-priority interface is disconnected, the system automatically switches to the next available egress.
- No manual switching or reconfiguration is required.
- The priority order configured by `netctl` is: `cellular > eth > onboard > wifi`

### Example

**Primary egress + automatic fallback**

```bash
netctl.sh setup_route --route "default>eth" --route "default>onboard" --route "default>wifi"
```

**Behavior:**
- Ethernet available → use Ethernet
- Ethernet disconnected → automatically switch to onboard
- Onboard unavailable → automatically switch to Wi‑Fi

### Recommended Typical Workflow

- Use `connect_wifi` to establish basic network connectivity.
- Configure a static IP or DHCP if needed.
- Use `setup_route` to configure the egress. If problems occur, use `setup_route --reset` to restore the system default behavior.

## Dependency Installation Tool

To help developers quickly and easily integrate and use third-party dependencies required by the main application in a non-root environment, Manifold 3 provides a built-in installation script, `install.py`, which can quickly download and install dependencies.

### Script Description

This script is used to install software without root privileges (rootless) and provides two installation methods: `apt` and `deb`.

- **apt**: Resolves and installs packages based on APT sources.
- **deb**: Installs packages from `.deb` files or URLs.

**Notes:**
- When using the `apt` subcommand to install software, custom APT sources and GPG keys are supported.
- Custom data and database storage directories are supported. The default path is `/open_app/install`.

### Runtime Requirements

- Python 3.7 or later
- Network access available (to access APT sources or key / `.deb` URLs)

### Usage Instructions

- **View help**

```bash
install.py -h
```

- **View help for the `apt` subcommand**

```bash
install.py apt -h
```

- **View help for the `deb` subcommand**

```bash
install.py deb -h
```

The command format is as follows:

```bash
install.py [sub_command(apt/deb)] [params ...]
```

### Parameter Description

#### apt Installation

**Subcommand:**

```bash
install.py apt [package …] [--source/-s <source> …] [--gpg-key/-k <key> …] [--dir/-d <directory>]
```

Examples:<br><br>APT source as file path (string format recommended):<br>`--source "/open_app/test/emqx_emqx-enterprise5.list"`<br><br>APT source as string (the whole line must be quoted):<br>`--source "deb https://mirrors.aliyun.com/ros2/ubuntu/ focal main"` |

### Common Issues and Troubleshooting

1. **Package not found** (`N: Unable to locate package <pkg>`)  
   Confirm that the `--source` parameter is correct and that the package is actually included in the specified source.

2. **GPG error** (`NO_PUBKEY ... / is not signed`)  
   - Confirm that the correct key is used with `--gpg-key`. It is recommended not to use `signed-by` in the `--source` repository line, because the script handles that automatically.
   - If a key file is used, make sure the key is readable and the path is correct.

3. **Network / certificate issues** (`Unable to establish SSL connection` or `Failed to fetch`)  
   Use `curl -v <URL>` or `wget --spider <URL>` to verify connectivity.
