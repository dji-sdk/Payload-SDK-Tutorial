## Application Packaging Process
Packaging a DPK application involves using `build_dpk.sh` DJI packaging tool to convert a development-state Manifold application package into a release-state installation package(xxx.dpk file). After generating the xxx.dpk file, it can be published for users to install and run. The overall process is as follows:

<img src="https://terra-1-g.djicdn.com/71a7d383e71a4fb8887a310eb746b47f/psdk/app_mgr/11.png" width=520 style="height:auto"/>
</div>

<br>For development-state application packages, they must contain the files listed in **Table 1: Development-State Application Package Files** (source code files, app.json, and CMakeLists.txt are mandatory, configuration files, resource files, and third-party library dependencies may not be required depending on the actual situation):

**Table 1** Development-State Application Package Files
| File Type | Description | Mandatory |
|-----------|------------|-----------|
| Source Code Files | Application source files (.h, .c, or .cpp) | Yes |
| Resource Files | Application resource files (currently not required, but may include icon files in the future) | No |
| Configuration Files | Dependency files needed during runtime, with paths specified in the `userconfig` field of app.json | No |
| Third-Party Library Files | Must be static libraries, dynamic libraries not supported | No |
| app.json Configuration File | Declares app name, version, description, etc. | Yes |
| CMakeLists.txt Build File | Compilation configuration file for packaging | Yes |

The release-state application package is the final Manifold application installation package(xxx.dpk file), containing executable files and configuration files.

## Packaging Tool Description
`build_dpk.sh` is used to generate xxx.dpk file. It must be called in the CmakeLists.txt file after compiling the executable. The tool is located at `tools/build_dpk/build_dpk.sh`. Usage:

```bash
# Usage
build_dpk.sh -i input app.json file
    [-o output dpk file path]
    [-h help]
Options:
   -h        Show help
   -i        Specify input app.json
   -o        Specify output .dpk path
```

## <a name="app-json-configuration">app.json Configuration File</a>

### app.json Description
The app.json file contains many fields. Below are the descriptions and specifications for each field:

| Property Name                  | Description | Data Type          | Optional  |
| :---------------------------- | :---------- | :----------------- | :-------- |
| `user_app_id`                 | The APP ID generated in the developer center, serving as the unique identifier for the application. Must match the `USER_APP_ID` in your code. Mismatch will cause installation failure. | String | Required |
| `firmware_version`            | Application version number. Format: `xx.xx.xx.xx` where each segment ranges 0-99. Must match `firmwareVersion` in code.<br>**Version segments**:<ul><li>Major (significant changes/new features)</li><li>Minor (new features/major fixes)</li><li>Debug</li><li>Modify (bug fixes)</li></ul> | String | Required |
| `is_ai_rendering`             | Whether the app performs AI rendering on pilot. Values: `true`/`false`. Default: `true` | String | Optional (default: true) |
| `platform`                    | Target platform. Supported value:<ul><li>`manifold3` (for Manifold 3 devices) </li></ul>| String | Required |
| `name_cn`                     | Chinese app name (max 47 bytes). Should reflect functionality and avoid duplication. Displayed in Chinese PILOT UI. | String | Required |
| `name_en`                     | English app name (max 32 bytes). Allowed chars: letters, numbers, hyphen (-), and dot (.). Must start with alphanumeric. And it cannot be "null". Displayed in non-CN/JP/FR PILOT UI. | String | Required |
| `name_jp`                     | Japanese app name (max 47 bytes). Displayed in Japanese PILOT UI. | String | Required |
| `name_fr`                     | French app name (max 47 bytes). Displayed in French PILOT UI. | String | Required |
| `description_cn`              | Chinese description (max 367 bytes). Format as JSON array for multi-line, Displayed in Chinese PILOT UI | String | Required |
| `description_en`              | English description (max 367 bytes, cannot be "null"). Format as JSON array for multi-line, Displayed in non-CN/JP/FR PILOT UI| String | Required |
| `description_jp`              | Japanese description (max 367 bytes). Format as JSON array for multi-line, Displayed in Japanese PILOT UI. | String | Required |
| `description_fr`              | French description (max 367 bytes). Format as JSON array for multi-line, Displayed in French PILOT UI. | String | Required |
| `maintainer_cn`               | Chinese developer info (max 127 bytes). Displayed in Chinese PILOT UI. | String | Required |
| `maintainer_en`               | English developer info (max 127 bytes, cannot be "null"). Displayed in non-CN/JP/FR PILOT UI. | String | Required |
| `maintainer_jp`               | Japanese developer info (max 127 bytes). Displayed in Japanese PILOT UI. | String | Required |
| `maintainer_fr`               | French developer info (max 127 bytes). Displayed in French PILOT UI. | String | Required |
| `ver_min`                     | Minimum required Manifold3 firmware version (format: `xx.xx.xx.xx`). App won't install if device firmware is lower. | String | Required |
| `ver_max`                     | Maximum supported Manifold3 firmware version (format: `xx.xx.xx.xx`). App won't install if device firmware is higher. | String | Required |
| `bin`                         | Relative path to executable (relative to `app.json`). File must exist. | String | Required |
| `userconfig`                  | Relative path to config files (relative to `app.json`). Optional if no config files exist. | String | Optional |

### 2. Create app.json
Refer to the app.json sample file and app.json description to create app.json. For details, see the chapter <a href = "#app-json-configuration">app.json Configuration File</a>.

### 3. Update CmakeLists.txt
First modify the top-level CMakeLists.txt file, changing:

```cmake
add_subdirectory(samples/sample_c/platform/linux/manifold2)
add_subdirectory(samples/sample_c++/platform/linux/manifold2)
```

to:

```cmake
add_subdirectory(samples/sample_c/platform/linux/manifold3)
add_subdirectory(samples/sample_c++/platform/linux/manifold3)
```

Then refer to `samples/sample_c/platform/linux/manifold3/CMakeLists.txt`, add following contents in app-level CMakeLists.txt file:

```cmake
# Set app.json and build_dpk.sh paths
set(JSON_FILE ${CMAKE_CURRENT_SOURCE_DIR}/app_json/app.json)
set(SCRIPT_FILE ${CMAKE_CURRENT_SOURCE_DIR}/../../../../../tools/build_dpk/build_dpk.sh)

# Set output dpk path
set(OUTPUT_PACKAGE ${CMAKE_BINARY_DIR}/dpk)

# Generate .dpk package
add_custom_command(TARGET ${PROJECT_NAME}
    POST_BUILD # Generate package after make
    COMMAND bash ${SCRIPT_FILE} -i ${JSON_FILE} -o ${OUTPUT_PACKAGE}
    DEPENDS ${JSON_FILE}
    COMMENT "Running build_dpk.sh to generate .dpk package from app.json"
)
```
### 4. Generate xxx.dpk
The PSDK supports CMake compilation, and xxx.dpk will be generated directly after compilation, use following commands:

```bash
cd Payload-SDK
mkdir build
cd build
cmake ..
make
```

After successful execution, the xxx.dpk package will be available at the `OUTPUT_PACKAGE` path, ready for installation and execution on Manifold devices.

### Application Installation Testing
After obtaining the DPK package, install/update it on the Manifold device. After successful installation, a directory will be created in /open_app partition, This directory contains all application files, and it will update automatically with app update.

To test the dpk app, you can start/stop the app via PILOT APP, and check whether the app runtime behavior meets expectations. if not, you can check application logs.

## FAQ
**1. How to fill in userconfig filed in app.json? How does the DPK app use packaged config files?**

Application packaging and runtime execution are decoupled processes.

Packaging Phase:
Application packaging organizes configuration files, executables, and other resources into a `xxx.dpk` installation package. The `userconfig` field in `app.json` specifies configuration file paths (relative to `app.json`) to include these files in the package. Paths may reference either: Entire directories, Specific files, Or a combination of both.

Runtime Phase:
After successful installation of the `.dpk` package, the Manifold3 device creates an application directory under `/open_app`. The application accesses files in this directory during execution.

Example Scenario:
In the app source code, the configuration files located at `/cfg/1/1.cfg`, `/cfg/2/2.cfg` , `/3/3.cfg`, and the app.json located at `/json/app.json`. Then valid `userconfig` formats and app runtime path include:

| `userconfig` Format | `userconfig` Value |Runtime Access Paths |
|---------------------|---------------------|---------------------|
| Directory-only      | `["../cfg", "../3"]` | `"1/1.cfg"`, `"2/2.cfg"`, `"3.cfg"` |
| File-specific       | `["../cfg/1/1.cfg", "../cfg/2/2.cfg", "../3/3.cfg"]` |`"1.cfg"`, `"2.cfg"`, `"3.cfg"` |
| Hybrid              | `["../cfg", "../3/3.cfg"]` |`"1/1.cfg"`, `"2/2.cfg"`, `"3.cfg"` |

**2. Installation fails with error: "Failed to start application service. Restart device and try again"**

**Solutions:**

- check app source code, to ensure all dependencies are static libraries (dynamic libraries unsupported) and the executable file can run normally.
- Check related logs at /blackbox/system/app_temp_files to investigate whether the application start normally and the reasons for any failure to start.
- If issues persist after fixing code, reboot device or contact DJI support.

**3. Installation fails with error: "Failed to verify application information. Make sure application information complies with specification"**

**Solutions:**

- Check whether the user_app_id and firmware_version fields in the app.json file comply with the specifications, refer to app.json description for details.
- If the values of the aforementioned fields are consistent, it is necessary to further check whether the application started normally without errors during installation. The startup logs during application installation can be found in the /blackbox/system/app_temp_files directory on the Manifold3 device.

**4. After the application is installed successfully, it fails to start, or although it starts successfully, the application's behavior does not match expectations:**

**Solutions:**
- Please ensure that all dependency files required by the application are packaged into the xxx.dpk file, and that the file read/write paths in the code are correct. For the correct read/write paths of dependency files, refer to the solution in: `Question 1. How to fill in userconfig filed in app.json? How does the DPK app use packaged config files?`
- If the application's dependency file read/write paths are correct and all dependencies are packaged into the xxx.dpk file, you can examine the application's runtime logs to debug reasons for startup failure or startup success with unexpected behavior.

**5.  The application depends on third-party libraries, and a permission issue is reported when installing the third-party library**

**Solutions:**

- Root accounts are not supported. Please include the dependency library in your application as a static library or source code.
